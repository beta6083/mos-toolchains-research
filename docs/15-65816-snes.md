# 15 — 65816 / SNES support via LLVM-MOS (research note)

The SNES CPU (Ricoh 5A22) is a WDC **65C816** — the 16-bit superset of the 6502.
exp 14 shows all four frontends *accept* `-mcpu=mosw65816`; this note answers what that
actually buys you, and how SNES homebrew is done on llvm-mos today. Backed by the
`mosw65816` codegen output, the backend TableGen, and the working zig-mos SNES SDK.

## One-line answer

llvm-mos has a **complete 65816 assembler / `lld`**, so real `.sfc` ROMs can be built and
shipped — but the **code generator emits 8-bit, 6502-style code**, with no native
16-bit-register mode, no 24-bit addressing, and no automatic banking. SNES support is a
homebrew/SDK achievement, **not** a compiler feature.

| layer | status |
|--|--|
| 65816 assembler / `llvm-mc` / `lld` (opcodes, long addr, relocs) | ✅ complete |
| native 16-bit codegen (`rep`/`sep`, 16-bit A/X/Y) | ❌ open — llvm-mos#321 |
| 24-bit address space / long pointers | ❌ open — llvm-mos#320 (pointers stay 16-bit) |
| automatic banking (bank trampolines) | ❌ open — llvm-mos#319 (done by hand via linker scripts) |
| official SNES SDK platform | ❌ none; draft PR #415 held back |
| shipping a real `.sfc` | ✅ yes — Celeste demake; zig-mos-examples SNES SDK |

## `-mcpu=mosw65816` = extra opcodes, not 16-bit codegen

A 16-bit add compiled for `mosw65816` is **byte-identical to mos6502** — 8-bit-chained
`adc` across the A/X pair, no 16-bit-accumulator mode:

```
$MOSCLANG --target=mos -mcpu=mosw65816 -Os -c add16.c   # unsigned short add16(a,b){return a+b;}
<add16>:  clc · adc $0 · tay · txa · adc $0 · tax · tya · rts   # identical to -mcpu=mos6502
```

The compiler **never emits `rep`/`sep`/`xce`/`jsl`/`mvn`** (verified: 0 occurrences in any
`mosw65816` codegen). The only gain over base 6502 is the extra 65C02/'816 addressing
modes (e.g. `sta ($zp)` zero-page-indirect).

**Why, at the source level** (`llvm/lib/Target/MOS/MOSDevices.td`): `FeatureW65816` is
defined purely as *"The new and modified instructions present on WDC 65816"* on top of the
65C02 — there is **no** 16-bit-register feature, no M/X-flag feature, no native-mode/banking/
address feature, and the register file has **only 8-bit working registers** (A/X/Y + the
6502 imaginary zero-page pointer pairs). `rep`/`sep`/`xce`/long-addressing opcodes exist but
only under `Predicates = [HasW65816]` so the **assembler / inline-asm** can encode them —
there are **no instruction-selection patterns** that make the compiler *choose* 16-bit mode.

## How SNES homebrew is actually done (zig-mos-examples)

`kassane/zig-mos-examples` ships a working SNES SDK over llvm-mos (via zig-mos):

- a Zig **`mos-snes` target** (`cpu_arch=.mos, os_tag=.snes`);
- a hand-written **`crt0.s`** that does the real **`clc; xce` emulation→native switch**, then
  keeps **M=1 (8-bit accumulator)** so llvm-mos's 8-bit codegen stays valid, using `rep #$30`
  (16-bit A/X/Y) only in hand-asm (NMI save, joypad auto-read). This is exactly what
  llvm-mos#320 predicts a minimal native target needs — *"adjusting crt0 to initialize the
  stack pointer to $01FF"* — i.e. run 6502-style code in native mode;
- **LoROM / HiROM / FastROM linker scripts** that place the 32-byte SNES ROM header ($FFC0)
  and native vectors ($FFE4) — the same linker-symbol idiom as exp 25's iNES header — doing
  the **banking by hand** (since #319 isn't implemented);
- **`sneslib`** (PPU / VRAM / DMA→CGRAM/OAM / joypad) and real `.sfc` demos: backdrop,
  colour-cycle, Zig-logo (CHR tiles), controller `pads`, a ~900-digit π-spigot.

Independent proof the compiler path ships: the **SNES Celeste demake** (SNESDEV-2025) builds
via llvm-mos and calls it *"the shipping path"* (full speed, LoROM-conformant) over five
other 65816 C compilers — the draft SDK target behind PR #415.

## vs. the established SNES toolchains

The native-65816 C path is **PVSnesLib** (`816-tcc` + `816-opt` + **WLA-DX**), plus vbcc /
Calypsi / WDCTools. **cc65 emits only 6502** (its `ca65` *assembles* 65816). Two *unrelated*
LLVM forks — `Peppar/llvm-C65`, `luizperes/llvm-to-snes` — do native-65816 LLVM codegen →
WLA-DX → `.sfc`, but they are **not** llvm-mos. So llvm-mos is the newcomer: best-in-class
assembler/linker + one shipped game, but no native-816 codegen and no upstream SNES platform.
